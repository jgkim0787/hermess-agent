모듈 1

LLM 다루기

개념

M0에서 우리는 LLM이 stateless 함수(텍스트 in → 텍스트 out, 기억 없음)임을 배웠고, 같은 질문을 두 번 보냈더니 "내 이름?"을 못 맞히는 것을 직접 봤다. 이번 모듈은 바로 그 한계의 해결이다. "기억"은 모델 안에 쌓이는 게 아니라 우리가 매 호출에 전체 대화 history를 다시 넣어주기 때문에 생긴다. 이 한 문장을 코드로 만드는 것이 M1의 전부다. 그 과정에서 호출의 입출력 단위(token), 대화 로그의 구조(messages), persona를 거는 슬롯(system), 응답이 멈추는 이유(stop_reason), 그리고 토큰을 흘려 출력하는 streaming까지 짚는다.

이 모듈의 running example은 한 줄이다: history를 직접 들고 다니는 멀티턴 대화 — 먼저 이름을 말하고, 다음 턴에 "내 이름?"으로 기억을 확인하고, SYSTEM persona로 말투를 바꾸고, 답을 streaming으로 흘려 출력한다. 개념·실습·클론 세 섹션이 모두 이 시나리오와 같은 식별자 (client, MODEL, messages, SYSTEM, chat, stream_chat)를 쓴다.

아래 개념 섹션의 코드블록은 핵심만 보인 발췌다(import·client 초기화·상수 정의는 생략). 그대로 복붙해 돌아가는 완전한 실행본은 직접 사용해보기의 step1~3에 있다.

token — 텍스트의 최소 단위이자 과금 단위

정의 — token: 모델이 텍스트를 다루는 최소 단위(대략 단어 조각). 입력·출력 token 수가 곧 과금·속도의 단위다.

비유하면 문자열을 잘게 쪼갠 청크 단위다. 네트워크 전송량이 곧 요금이듯, 모델도 주고받은 token "양"이 곧 비용이다. 코스 기본 모델 claude-sonnet-4-6은 input $3 / output $15 per 1M tokens다. 이 비용 감각이 중요한 이유는, 멀티턴이 되는 순간 우리가 매번 전체 history를 다시 보내기 때문에 대화가 길어질수록 입력 token이 빠르게 누적된다는 점이다. (긴 대화를 줄이는 압축은 M4의 동기다.)

토큰을 정확히 세려면 client.messages.count_tokens(model=..., messages=...)를 쓴다. (Claude 전용이 아니므로 tiktoken은 쓰지 않는다.)

messages — role을 가진 append-only 대화 로그

정의 — message: 대화의 한 발화. role(user/assistant)과 content를 가지며, 리스트로 쌓이는 append-only 로그다. 첫 항목은 반드시 user.

request/response가 번갈아 쌓이는 append-only 이벤트 로그라고 보면 된다. 우리가 사용자 발화를 user로 append하고, 모델의 답을 assistant로 append한다. 이 로그 자체가 "대화 history"이고, M0과 달리 이번엔 우리가 이 리스트를 직접 들고 다닌다.

# messages 는 우리가 직접 관리하는 append-only 로그
messages = [
    {"role": "user",      "content": "내 이름은 정구야."},
    {"role": "assistant", "content": "반가워요, 정구님."},
    {"role": "user",      "content": "내 이름?"},          # 다음 호출은 이 리스트 통째로 보낸다
]

system prompt — 대화 바깥의 persona / 불변 규칙 슬롯

정의 — system prompt: 대화 내용과 별개로 모델의 persona·불변 규칙을 거는 최상위 슬롯 (system="..." 파라미터). messages 안에 넣는 게 아니다.

프로세스의 환경설정 / 불변 규칙에 가깝다 — messages 로그 바깥에 있는 전역 설정이다. 같은 대화(messages)라도 system만 바꾸면 말투(persona)가 달라진다. 그래서 running example의 "말투 바꾸기"는 SYSTEM 상수 한 줄만 교체하면 끝난다.

SYSTEM = "너는 간결한 한국어 도우미야."   # persona 전환 시 이 상수만 바꾼다

resp = client.messages.create(
    model=MODEL, max_tokens=1024,
    system=SYSTEM,          # ← messages 가 아니라 별도 슬롯
    messages=messages,
)

max_tokens 와 stop_reason — 얼마나 쓰나 / 왜 멈췄나

정의 — max_tokens: 한 응답에서 생성할 출력 token의 하드 상한. 도달하면 문장 중간이라도 끊긴다.

출력 버퍼의 상한(cap)이다. 너무 작게 잡으면 답이 중간에 잘리고, 그때 모델은 stop_reason="max_tokens"로 "상한 때문에 멈췄다"고 알려준다. 짧은 예제는 1024면 충분하다.

정의 — stop_reason: 모델이 왜 멈췄는지 알려주는 플래그. end_turn(자연 종료) / max_tokens(상한 도달) / tool_use(도구 호출 원함) / refusal(안전상 거부) — (서버 도구 재개용 pause_turn도 있다).

함수 반환의 exit code / 종료 사유라고 보면 된다. 이번 모듈에선 거의 end_turn만 보겠지만, 이 플래그를 읽는 습관이 중요하다 — M2에서 tool_use면 도구를 실행하고, M3의 agent loop는 stop_reason이 end_turn이 될 때까지 while로 반복한다. 즉 지금 배우는 stop_reason이 다음 두 모듈의 분기 조건이 된다.

★ stateless — "기억"은 우리가 history를 다시 보내기 때문

정의 — stateless: API 호출 자체는 이전 호출을 기억하지 못한다. 멀티턴이 되려면 매 호출에 전체 messages history를 다시 보내야 한다. (이 모듈의 핵심)

세션을 쥐지 않는 stateless HTTP 핸들러와 같다. 서버(모델)는 상태를 보관하지 않고, "기억"은 우리가 매번 첨부하는 history 덕분에 생긴다. M0에서 두 번째 질문이 첫 대화를 몰랐던 건 각 호출이 독립이었기 때문이다. 해결책은 단순하다 — 한 리스트(messages)에 발화를 누적하고, 매 호출에 그 리스트를 통째로 보낸다.

# cheatsheet "멀티턴 대화" 패턴. 세 섹션이 공유하는 chat() 시그니처.
messages = []

def chat(user_text: str) -> str:
    messages.append({"role": "user", "content": user_text})     # 1) 발화 누적
    resp = client.messages.create(
        model=MODEL, max_tokens=1024,
        system=SYSTEM,
        messages=messages,                                       # 2) history 통째로 재전송
    )
    answer = next((b.text for b in resp.content if b.type == "text"), "")
    messages.append({"role": "assistant", "content": answer})    # 3) 답도 다시 누적
    return answer

chat("내 이름은 정구야.")
chat("내 이름?")            # ← 같은 messages 를 다시 보냈으므로 이번엔 "정구"라고 답한다
왜 이번엔 기억할까? 두 번째 chat("내 이름?") 호출에는 첫 발화와 그 답이 messages에 그대로 들어 있다. 모델이 매번 받는 입력 안에 "정구"가 있으니 답할 수 있는 것이다. "기억"은 모델의 상태가 아니라 우리가 다시 보내는 history다 — M0의 "내 이름?" 실패와 정확히 같은 질문으로 이 차이를 확인한다. 첫 항목이 반드시 user여야 하고(아니면 400), assistant 응답을 append하지 않으면 다음 턴에 또 잊는다는 점만 주의한다.

streaming — 토큰을 generator처럼 순차 수신

정의 — streaming: 응답이 완성될 때까지 기다리지 않고 token을 생성되는 대로 순차 수신하는 방식.

chunked transfer, 혹은 generator(yield)를 떠올리면 된다. 토큰을 한 조각씩 받아 즉시 화면에 찍으므로 긴 답이 "타이핑되듯" 흘러나온다. client.messages.stream(...)을 with로 열고 stream.text_stream을 순회한다.

with client.messages.stream(
    model=MODEL, max_tokens=1024,
    system=SYSTEM, messages=messages,
) as stream:
    for text in stream.text_stream:        # 토큰 조각이 도착하는 대로
        print(text, end="", flush=True)    # 즉시 출력 (generator 순회처럼)
    final = stream.get_final_message()     # 완료 후 전체 메시지 (usage·stop_reason 등)
비유의 한계: "generator"는 순차 수신이라는 직관엔 맞지만, text_stream이 내주는 조각은 한 token 단위로 딱 맞아떨어지는 게 아니다(SDK가 텍스트 델타를 모아 준다). "완성을 기다리지 않고 조금씩 받는다"는 점만 가져가면 된다.
다음 단계 예고: 매 호출에 history를 다시 보내니 token이 계속 쌓인다는 점은 이미 봤다. "history가 곧 모델에 주는 context"이며, 그 한도(context window)와 비용을 본격적으로 다루는 것은 M4(Memory & Context)다.

한눈 정리

  • token = 텍스트의 최소 단위이자 과금 단위. sonnet-4-6 = $3/$15 per 1M(in/out) — history가 길수록 비용↑.
  • messages = role(user/assistant)을 가진 append-only 로그. 우리가 직접 들고 다닌다. 첫 항목은 user.
  • system prompt = 대화 바깥의 persona/불변 규칙 슬롯. SYSTEM 한 줄만 바꿔 말투를 바꾼다.
  • max_tokens = 출력 하드 상한, stop_reason = 왜 멈췄나(end_turn/max_tokens/tool_use/refusal) → M2·M3 분기 조건.
  • ★ stateless = API는 무상태. "기억"은 우리가 매 호출에 전체 messages를 다시 보내기 때문. streaming으로 답을 조각조각 흘려 출력한다.

직접 사용해보기

아래 코드를 직접 타이핑해 실행해 보세요. 전제: Python 3.10+, 그리고 Anthropic API 키. 각 단계는 한 파일(stepN.py)로 그대로 복붙·실행되도록 import와 client 초기화를 포함합니다.

0) 준비

pip install anthropic
export ANTHROPIC_API_KEY="sk-ant-...";  # 본인 키로 (코드에 하드코딩 금지)
Module 0과 동일하다. 키는 console.anthropic.com에서 발급한다. 멀티턴은 매 호출에 history 전체를 다시 보내 토큰이 빠르게 늘므로, 실습은 작은 max_tokens로 한다.

1) 멀티턴 — history를 직접 들고 "내 이름?"으로 기억 확인

Module 0에서 본 stateless의 해결책이다. messages 리스트에 user/assistant 발화를 append-only로 쌓고, 매 호출에 그 리스트를 통째로 보낸다. 그래서 두 번째 턴에서 "내 이름?"을 물어도 모델이 기억한다.

# step1.py
import anthropic

client = anthropic.Anthropic()          # ANTHROPIC_API_KEY 환경변수 자동 사용
MODEL = "claude-sonnet-4-6"             # 코스 기본 모델
SYSTEM = "너는 간결한 한국어 도우미야."   # persona/불변 규칙 (대화 로그 바깥의 슬롯)

messages = []                           # append-only 대화 로그 (첫 항목은 반드시 user)

def chat(user_text: str) -> str:
    messages.append({"role": "user", "content": user_text})   # 1) 내 발화 누적
    resp = client.messages.create(
        model=MODEL,
        max_tokens=200,
        system=SYSTEM,                  # system은 messages 안이 아니라 별도 파라미터
        messages=messages,              # 2) ★ 매 호출에 history 전체를 다시 보낸다
    )
    answer = next((b.text for b in resp.content if b.type == "text"), "")
    messages.append({"role": "assistant", "content": answer})  # 3) 모델 답도 누적
    return answer

print(chat("내 이름은 정구야. 기억해줘."))
print("---")
print(chat("내 이름?"))                 # history를 다시 보냈으니 기억한다
print("---")
print("history 길이:", len(messages))   # user/assistant 4개 누적
python step1.py

왜 이번엔 기억할까? 모델 안에 상태가 쌓인 게 아니다. 두 번째 호출의 messages에 첫 턴(이름)이 그대로 들어 있어서다. API는 stateless — "기억"은 우리가 매 호출에 messages를 통째로 첨부하기 때문에 생긴다. Module 0의 "독립 호출 2번 → 못 기억"과 정확히 반대다.

2) 말투 바꾸기 — SYSTEM(persona) 상수만 교체

대화 내용은 그대로 두고 SYSTEM 한 줄만 바꾼다. 같은 질문에 말투(persona)가 달라진다. persona는 messages 로그가 아니라 별도 슬롯이라, 이 상수만 교체하면 된다.

# step2.py
import anthropic

client = anthropic.Anthropic()
MODEL = "claude-sonnet-4-6"

def ask_with_persona(system_text: str, user_text: str) -> str:
    resp = client.messages.create(
        model=MODEL,
        max_tokens=120,
        system=system_text,             # ← 이 슬롯만 바꿔 persona 전환
        messages=[{"role": "user", "content": user_text}],
    )
    return next((b.text for b in resp.content if b.type == "text"), "")

question = "오늘 날씨 좋네. 한마디 해줘."
print("[공손체]", ask_with_persona("너는 아주 공손한 비서야. 존댓말을 쓴다.", question))
print("[해적]  ", ask_with_persona("너는 바다를 누비는 해적이야. 거칠게 반말로 말한다.", question))

[해적]   하! 이런 날엔 돛 펴고 바다로 나가야지. 뭘 꾸물대냐!
출력은 매번 조금씩 다르다. LLM 출력은 확률적이라 문장은 달라질 수 있다 — 중요한 건 말투가 SYSTEM에 따라 바뀐다는 점이다. Hermes에서는 이 문자열을 SYSTEM 상수 한 곳에 두고 바꾼다.

3) streaming — 토큰을 생성되는 대로 흘려 출력

답이 완성될 때까지 기다리지 않고, 토큰을 generator처럼 한 조각씩 받아 즉시 출력한다. client.messages.stream(...)의 text_stream을 순회한다. 멀티턴과 합쳐, stream_chat도 답변을 history에 다시 누적한다.

# step3.py
import anthropic

client = anthropic.Anthropic()
MODEL = "claude-sonnet-4-6"
SYSTEM = "너는 간결한 한국어 도우미야."

messages = []

def stream_chat(user_text: str) -> str:
    messages.append({"role": "user", "content": user_text})
    answer = ""
    with client.messages.stream(
        model=MODEL,
        max_tokens=200,
        system=SYSTEM,
        messages=messages,              # 멀티턴과 동일: history 전체를 보낸다
    ) as stream:
        for text in stream.text_stream: # 토큰이 도착하는 대로
            print(text, end="", flush=True)
            answer += text
    print()                             # 줄바꿈
    messages.append({"role": "assistant", "content": answer})  # 답도 누적
    return answer

stream_chat("내 이름은 정구야.")
stream_chat("내 이름이 뭐라고 했지? 한 문장으로.")   # 스트리밍 + 기억 둘 다 확인
python step3.py
알겠어요, 정구님!
정구님이라고 하셨어요.
출력이 한 글자씩 흘러나오면 streaming이 동작하는 것이다(위 결과는 최종 화면 모습). 긴 출력이나 큰 max_tokens에서 특히 유용하다 — 다음 단계 Hermes의 REPL이 이 stream_chat으로 응답을 흘려 보여준다.
막히면:
  • AuthenticationError(401) → ANTHROPIC_API_KEY 미설정/오타. echo $ANTHROPIC_API_KEY로 확인.
  • NotFoundError(404) → 모델 ID 오타. claude-sonnet-4-6 그대로(날짜 suffix 붙이지 말 것).
  • BadRequestError(400) "first message must use the 'user' role" → 첫 메시지가 assistant다. messages는 반드시 user로 시작한다. assistant 응답만 append하고 user를 빠뜨리지 않았는지 확인.
  • 이름을 기억 못 함 → 모델 답을 messages에 append하지 않았거나, 매 턴 새 리스트를 만들고 있다. chat이 같은 messages 리스트에 계속 누적하는지 확인(stateless라 history가 곧 기억).
  • 연결 오류(APIConnectionError) → 네트워크/프록시 확인.

Hermes 만들기

이번 단계: M0의 "두뇌"(ask)에 멀티턴 + streaming + persona를 더한다. 개념↔코드 매핑 한 줄 — 개념의 stateless / history = 아래 chat(messages)의 messages 인자(매 호출에 통째로 다시 보냄), persona = SYSTEM 상수, streaming = stream_chat의 stream.text_stream.

전제: M0에서 만든 clone-project/hermes/(config.py, llm.py의 ask, 단발 __main__.py) 위에 더한다. 이번엔 llm.py에 함수 2개를 추가하고 __main__.py만 REPL로 교체한다. ask는 지우지 않고 남겨 둔다. 참고로 ask는 M0 동작을 그대로 보존하려고 SYSTEM을 붙이지 않고, 이번에 추가하는 chat/stream_chat만 SYSTEM persona를 적용한다.

1) hermes/llm.py 에 SYSTEM 상수 추가

지금 무엇을/왜: persona(말투·불변 규칙)는 대화 로그가 아니라 system 슬롯으로 건다. 이 한 줄 상수만 바꾸면 Hermes의 성격이 바뀐다. llm.py 상단, client = ... 줄 바로 아래에 추가한다.

# hermes/llm.py  (기존 import / client 아래에 추가)
SYSTEM = "너는 Hermes라는 간결한 한국어 도우미야. 군더더기 없이 핵심만 답한다."

2) chat(messages) 추가 — history를 인자로 받기

지금 무엇을/왜: 멀티턴의 핵심은 stateless다 — API는 이전 호출을 기억 못 하니 매번 전체 messages를 다시 보내야 한다. M0의 ask는 메시지를 안에서 만들었지만, chat은 history 리스트를 인자로 받아 그대로 보낸다(REPL이 history를 들고 다니게 하려고). ask 함수 아래에 추가한다.

# hermes/llm.py  (ask 함수 아래에 추가)
def chat(messages: list) -> str:
    """history(messages 리스트)를 받아 호출하고, 답변 텍스트를 반환한다.
    stateless API라 매 호출에 전체 history를 다시 보낸다 — 이 messages가 곧 '기억'."""
    resp = client.messages.create(
        model=MODEL,
        max_tokens=MAX_TOKENS,
        system=SYSTEM,            # persona는 messages가 아니라 별도 슬롯
        messages=messages,        # 전체 history를 통째로 전송
    )
    return "".join(b.text for b in resp.content if b.type == "text")
실습과의 차이 한 줄: 실습의 chat(user_text)는 모듈 전역 messages를 쓰는 편의형이었다. Hermes의 chat(messages)는 history를 인자로 받아 더 재사용 가능하게 만든 형태다 (REPL이 자기 messages를 들고 와 넘긴다).

3) stream_chat(messages) 추가 — 토큰 스트리밍

지금 무엇을/왜: chat은 답이 다 만들어질 때까지 기다린다. stream_chat은 token을 generator(stream.text_stream)처럼 받아 생성되는 대로 출력한다 — REPL이 즉시 반응하게. 출력하면서, 나중에 history에 append할 수 있도록 모은 전체 텍스트를 반환한다. chat 아래에 추가한다.

# hermes/llm.py  (chat 함수 아래에 추가)
def stream_chat(messages: list) -> str:
    """history를 받아 token을 흘려 출력하고, 완성된 전체 답변 텍스트를 반환한다."""
    chunks = []
    with client.messages.stream(
        model=MODEL,
        max_tokens=MAX_TOKENS,
        system=SYSTEM,
        messages=messages,
    ) as stream:
        for text in stream.text_stream:   # token이 오는 대로
            print(text, end="", flush=True)
            chunks.append(text)
    print()                                # 줄바꿈 마무리
    return "".join(chunks)                 # history에 append 하려고 모아서 반환

4) hermes/__main__.py 를 REPL로 교체

지금 무엇을/왜: M0의 단발 진입점(sys.argv 한 번 실행)을 REPL로 갈아끼운다. 핵심은 messages 리스트를 하나 들고, 매 입력을 {"role":"user", ...}로 append → 스트리밍 출력 → assistant 답변을 다시 append 하는 것. 이 누적이 곧 "기억"이다(stateless의 해결). 파일 내용을 아래로 통째 교체한다(M0의 ask import는 더 안 쓴다).

# hermes/__main__.py  (M0 단발 버전을 아래로 교체)
from .llm import stream_chat

def main():
    print("Hermes REPL — 빈 입력 또는 'exit'로 종료\n")
    messages = []                          # append-only 대화 history (이게 곧 '기억')
    while True:
        user = input("you> ").strip()
        if user == "" or user == "exit":   # 종료 조건
            print("bye.")
            break
        messages.append({"role": "user", "content": user})
        print("hermes> ", end="", flush=True)
        answer = stream_chat(messages)     # 전체 history를 보내고 token을 흘려 출력
        messages.append({"role": "assistant", "content": answer})  # 답도 history에 누적

if __name__ == "__main__":
    main()
왜 둘 다 append 하나? user 입력만 넣고 assistant 답을 안 넣으면, 다음 턴에 모델은 자기가 방금 뭐라 답했는지 모른다. messages는 user/assistant가 번갈아 쌓이는 append-only 로그라, 양쪽 다 넣어야 다음 호출에서 온전한 맥락이 된다.

이름을 말한 뒤, 다음 턴에 "내 이름?"으로 기억을 확인한다(M0에선 못 기억했던 바로 그 질문).

you> 내 이름은 정구야.

왜 이번엔 기억할까? 두 번째 입력 "내 이름?" 차례에 messages에는 이미 1턴의 user/assistant가 들어 있다. 그 history를 통째로 다시 보냈기 때문에 모델이 답할 수 있다 — "기억"은 모델이 아니라 우리 messages 리스트에 있다. 이게 stateless API로 멀티턴을 만드는 전부다.
지금 Hermes의 한계 = 다음 모듈의 동기: 이제 대화는 기억하지만, 아직 손발(도구)이 없다 — 파일을 읽거나 명령을 실행하지 못하고 "말"만 한다. 다음 모듈(M2 · 첫 도구)에서 tool use로 Hermes가 실제 함수를 호출하게 만든다. 그때 stop_reason(이번에 본 그 플래그)이 tool_use인지 보고 분기하는 게 시작점이다.
막히면: messages의 첫 항목이 assistant면 400 → 첫 메시지는 반드시 user(REPL은 user 입력으로 시작하니 OK). 기억을 못 하면 assistant 답을 history에 append 안 했는지 확인(4단계의 두 번째 append). AuthenticationError(401)/NotFoundError(404)는 키·모델 ID 문제로 M0 "막히면"과 동일 (ANTHROPIC_API_KEY 설정, 모델은 claude-sonnet-4-6 그대로).
